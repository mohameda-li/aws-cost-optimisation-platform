Security

A customer-owned model that keeps infrastructure analysis inside the customer account.

The security story is deliberately simple: the code runs in the customer environment, reports stay in the customer bucket and logs remain in the customer AWS account. The platform coordinates deployment generation, but it does not become a central hosted engine for reading production infrastructure data.

Security posture Customer-controlled runtime
Lambda runner and optimisers execute in customer AWS
Reports are written to the configured customer S3 bucket
Operational logs remain in CloudWatch within the deployed account
No automated remediation actions are applied to infrastructure

Where the code runs

Runner and optimiser Lambdas execute inside the customer account. Reports are stored in the customer S3 bucket and logs are written to the customer CloudWatch environment. This makes the deployment easier to audit and remove.

Access model

The deployment uses a scoped IAM role to read service metadata, pricing inputs and reporting targets needed for optimisation. The system does not perform automated remediation actions against customer infrastructure.

Reporting and notifications

Generated reports are written as HTML and JSON files and can be delivered by email using the configured SMTP or SES settings. Notification emails are designed to point recipients back to the relevant customer workflow clearly.

Operational controls

Terraform supports straightforward destroy and redeploy workflows. Historical reports are retained in a controlled way, and the deployment can be regenerated from the latest onboarding data when customer settings change.
